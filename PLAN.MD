# PLAN.MD

# MVP магазина приложений на Android — декомпозиция

## Цель MVP
Реализовать версию магазина приложений на Android со следующим функционалом:
- список приложений;
- карточка приложения;
- установка и удаление приложений.

---

## Общая стратегия декомпозиции

Проект разбит на 4 эпика:
1. Базовая архитектура и инфраструктура
2. Список приложений
3. Карточка приложения
4. Установка и удаление приложений

Оценки указаны в рабочих днях.

---

# Epic 1. Подготовка проекта и базовая архитектура
**Оценка:** 4 дня

## Цель
Подготовить проектную основу, на которой можно реализовывать остальные фичи MVP.

## Задачи

### 1.1. Создать Android-проект и настроить базовую структуру модулей/пакетов
**Оценка:** 0.5 дня  
**Результат:** создан проект, определены пакеты `data`, `domain`, `presentation` или аналогичная структура.

### 1.2. Подключить необходимые зависимости
**Оценка:** 0.5 дня  
**Результат:** подключены Compose, Navigation, ViewModel, Coroutines, Retrofit/OkHttp, Coil.

### 1.3. Настроить навигацию между экранами
**Оценка:** 0.5 дня  
**Результат:** есть маршруты для списка приложений и карточки приложения.

### 1.4. Описать базовые domain-модели приложения
**Оценка:** 0.5 дня  
**Результат:** созданы модели, например `AppItem`, `AppDetails`, состояния установки.

### 1.5. Подготовить источник данных для MVP
**Оценка:** 1 день  
**Результат:** реализован моковый источник данных или локальный JSON/заглушка API со списком приложений.

### 1.6. Реализовать базовый repository для получения данных о приложениях
**Оценка:** 1 день  
**Результат:** UI получает данные не напрямую, а через repository/use case слой.

# Epic 2. Экран списка приложений
**Оценка:** 5 дней

## Цель
Показать пользователю список доступных приложений с базовой информацией и переходом в карточку приложения.

## Задачи

### 2.1. Реализовать use case получения списка приложений
**Оценка:** 0.5 дня  
**Результат:** есть отдельная логика получения списка приложений для presentation-слоя.

### 2.2. Реализовать ViewModel для экрана списка
**Оценка:** 1 день  
**Результат:** ViewModel загружает список, хранит состояние загрузки, успеха и ошибки.

### 2.3. Создать UI списка приложений
**Оценка:** 1 день  
**Результат:** отображается список приложений через `LazyColumn` или аналогичный компонент.

### 2.4. Создать UI элемента списка приложения
**Оценка:** 0.5 дня  
**Результат:** каждая карточка в списке показывает иконку, название, категорию/краткое описание.

### 2.5. Обработать состояния loading / error / empty
**Оценка:** 1 день  
**Результат:** пользователь видит корректный экран при загрузке, ошибке и пустом списке.

### 2.6. Добавить переход из списка в карточку приложения
**Оценка:** 0.5 дня  
**Результат:** по нажатию на элемент списка открывается экран деталей.

### 2.7. Провести ручное тестирование экрана списка
**Оценка:** 0.5 дня  
**Результат:** проверены отображение, навигация и базовые сценарии.

# Epic 3. Экран карточки приложения
**Оценка:** 4.5 дня

## Цель
Показать подробную информацию о выбранном приложении.

## Задачи

### 3.1. Реализовать use case получения деталей приложения
**Оценка:** 0.5 дня  
**Результат:** есть отдельная бизнес-логика для загрузки карточки приложения по ID.

### 3.2. Реализовать ViewModel для экрана карточки приложения
**Оценка:** 1 день  
**Результат:** ViewModel получает ID приложения и загружает детальные данные.

### 3.3. Создать UI карточки приложения
**Оценка:** 1 день  
**Результат:** экран показывает иконку, название, описание, рейтинг, размер, разработчика или другие основные поля.

### 3.4. Отобразить текущее состояние установки на карточке
**Оценка:** 0.5 дня  
**Результат:** пользователь видит, установлено приложение или нет.

### 3.5. Обработать состояния loading / error
**Оценка:** 0.5 дня  
**Результат:** экран корректно работает при загрузке и ошибке данных.

### 3.6. Провести ручное тестирование карточки приложения
**Оценка:** 1 день  
**Результат:** проверены загрузка деталей, корректность данных и возврат назад.

# Epic 4. Установка и удаление приложений
**Оценка:** 5.5 дня

## Цель
Добавить возможность установить приложение и удалить его, обновляя состояние в списке и карточке.

## Задачи

### 4.1. Спроектировать модель состояния установки
**Оценка:** 0.5 дня  
**Результат:** определены состояния, например `NotInstalled`, `Installing`, `Installed`, `Removing`.

### 4.2. Реализовать storage/источник состояния установленных приложений
**Оценка:** 1 день  
**Результат:** состояние установленных приложений сохраняется локально в памяти, DataStore или другой простой реализации для MVP.

### 4.3. Реализовать use case установки приложения
**Оценка:** 1 день  
**Результат:** по действию пользователя приложение переходит в установленное состояние.

### 4.4. Реализовать use case удаления приложения
**Оценка:** 1 день  
**Результат:** по действию пользователя приложение удаляется из списка установленных.

### 4.5. Добавить кнопки "Установить" / "Удалить" в карточку приложения
**Оценка:** 0.5 дня  
**Результат:** пользователь может менять состояние приложения с экрана деталей.

### 4.6. Обновить список приложений с учетом состояния установки
**Оценка:** 0.5 дня  
**Результат:** в списке видно, какие приложения уже установлены.

### 4.7. Обработать промежуточные состояния установки/удаления
**Оценка:** 0.5 дня  
**Результат:** UI показывает progress/loading во время установки и удаления.

### 4.8. Провести ручное тестирование сценариев установки и удаления
**Оценка:** 0.5 дня  
**Результат:** проверены основные пользовательские сценарии и синхронность состояний между экранами.

# Ожидаемый результат MVP

После выполнения всех эпиков пользователь может:
- открыть список приложений;
- перейти в карточку выбранного приложения;
- установить приложение;
- удалить установленное приложение;
- видеть актуальное состояние приложения и в списке, и в карточке.
